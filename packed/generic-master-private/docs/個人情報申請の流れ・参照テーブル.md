# 個人情報申請の流れ・参照テーブル

## データ取得

画面表示時は、まず申請保存テーブル (`staff_profile_request`) を参照する。

### 1. 申請中データの確認

対象スタッフ・対象データ種別(basic,address,bank等)の最新申請を取得する。

### 2. 取得した申請のステータスを判定

以下のステータスの場合は、その申請データを画面へ表示する。

| request_status | 内容   | 使用データ                 |
| -------------- | ---- | --------------------- |
| draft          | 下書き  | staff_profile_request |
| submitted      | 申請中  | staff_profile_request |
| returned       | 差し戻し | staff_profile_request |

以下のステータス、または申請データが存在しない場合は、正式データを基に新しい申請データを作成する。

| request_status | 内容   |
| -------------- | ---- |
| approved       | 承認済み |
| rejected       | 却下済み |
| レコードなし         | 初回申請 |

処理内容

1. `staff_profile_data`（正式保存テーブル）から現在有効なデータを取得
2. 取得した内容を `staff_profile_request` にコピー
3. 新しい申請レコードとして作成する
4. 作成した申請データを画面へ表示する

※ 新規作成時の `request_status` は `draft` とする。

# データ保存

画面で編集した内容は `staff_profile_request` に保存する。

保存時は JSON データと申請情報を更新する。

## 一時保存

ボタン：**一時保存**

更新内容

* data_json を更新
* request_status = `draft`
* updated_at を更新
* updated_by を更新

---

## 登録・変更申請

ボタン：**登録・変更申請**

更新内容

* data_json を更新
* request_status = `submitted`
* submitted_at を更新
* submitted_by を更新
* updated_at を更新
* updated_by を更新

---

## 考慮するとよい点

設計上、以下も仕様として決めておくと運用しやすくなります。

1. **申請中レコードは1件のみとするか**

   * 同一スタッフ・同一データ種別で `draft`・`submitted`・`returned` を同時に複数持てるのか。
   　→1件のみを想定

2. **正式データが存在しない初回登録**

   * 新規スタッフなど正式データが存在しない場合は、空データから `staff_profile_request` を作成する仕様にしておく。
   　→未定

3. **履歴管理**

   * `staff_profile_request` は削除せず履歴として保持するのか。(申請履歴を管理するか)
   　→現在は保持する設計になっている

4. **排他制御**

   * 管理者と本人が同時に編集した場合の扱い。
   　→管理者とスタッフ本人が扱うデータテーブルは違うので問題はないと考えられるが、管理者同士の場面について考える必要がある(?)
   * 楽観ロック（`updated_at` や `version`）を用いるかを決めておく。
   　→管理者など

5. **承認後の編集開始**

   * 承認済みデータを再度編集する際は、必ず正式データをコピーして新しい申請レコードを作成し、過去の申請レコードは変更しない。
