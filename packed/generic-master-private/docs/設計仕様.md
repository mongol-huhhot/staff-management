# マスタ設計  

## 方針
- マスタデータを適用日でトランザクションデータとして管理する
- 編集データ中データを本番データを別テーブルに管理する
- 承認、あるいは、確定した場合データを本番テーブルに反映する

## 承認範囲 
承認はお客様のご要望に応じてあり・なし設定ができる   
範囲は：   

```text
各申請テーブル
  staff_profile_request
  mtb_user_request
  contract_request
  bank_request
  dependent_request
        ↓
共通アクション履歴
  workflow_schema.request_action_log

```

## ユーザー、スタッフ情報の運用フロー

```text

mtb_user_request
  draft
    ↓ 一時保存
  submitted
    ↓
    承認必要あり  → mtb_usersへまだ反映しない
    承認不要      → mtb_usersへ即反映
    既存ユーザー修正 → mtb_usersへ即反映
  approved
    ↓
    承認必要な新規登録だけ mtb_usersへ反映

```

## ユーザー・スタッフ情報の連携

- 本人ユーザー → 自分のスタッフ情報
- 店長ユーザー → 複数スタッフを見る
- 人事ユーザー → 全スタッフを見る
- 代理入力者 → 一時的に特定スタッフを編集

```text

A. 本人登録
  ユーザー登録
  メール認証
  スタッフ情報入力
  m_staff_user 自動作成
  relation_status = pending
  （ ここまで、自分の入力内容を管理できる（新規（初回）・修正・削除（登録取消）））
  ↓
  会社側が追加情報入力・確認
  ↓
  承認不要なら active
  承認必要なら approved 後 active

B. 会社側登録（あまり使われない）
  スタッフ仮登録
  招待メール
  本人ユーザー登録
  メール認証
  m_staff_user 自動作成
  relation_status = pending
  ↓
  会社方針により承認
  ↓
  active

C. 例外対応
  管理者が手動関連付け
  relation_status = active

A, C をメインの検討します。

```

## 設計概要
- 個人情報ステータス： draft, submitted, approved, rejected, cancelled
- 承認は一次のみ
- 承認ある場合、approved時に編集データを本番データにコピー
- 承認なしの場合はsubmitted時に編集データを本番データにコピー
- rejected, cancelledは管理者の操作でステータスがdraft, submittedの場合のみできます
- cancelled時、再提出と

### 月内で有効だったスタッフ情報取得  

```sql
WITH params AS (
  SELECT
    DATE '2026-02-01' AS month_start,
    (DATE '2026-02-01' + INTERVAL '1 month - 1 day')::date AS month_end
),
hist AS (
  SELECT
    d.*,
    LEAD(d.valid_from) OVER (
      PARTITION BY d.staff_id
      ORDER BY d.valid_from
    ) - 1 AS valid_to
  FROM staff_schema.staff_profile_data d
)
SELECT h.*
FROM hist h
CROSS JOIN params p
WHERE h.valid_from <= p.month_end
  AND COALESCE(h.valid_to, DATE '9999-12-31') >= p.month_start
ORDER BY h.staff_code, h.valid_from

```
